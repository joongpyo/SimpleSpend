## 3. System Architecture (시스템 아키텍처)

SwiftUI에 최적화된 **MVVM(Model-View-ViewModel)** 패턴과 **SwiftData**를 결합한 구조입니다. 2명이서 작업할 때 코드가 엉키지 않도록 역할을 분리하기 좋습니다.

코드 스니펫

`graph LR
    subgraph "View (UI Layer)"
        V1[MainView: 리스트/합계]
        V2[AddExpenseView: 입력창]
    end

    subgraph "ViewModel (Logic Layer)"
        VM[ExpenseViewModel: 데이터 처리/연산]
    end

    subgraph "Model (Data Layer)"
        M[Expense Model: SwiftData]
        DB[(Local Storage)]
    end

    V1 <--> VM
    V2 <--> VM
    VM <--> M
    M <--> DB`

### 아키텍처 상세 설명

- **Model:** 지출 데이터의 규격을 정의합니다. (금액, 메모, 날짜)
- **View:** 사용자가 보는 화면입니다. 로직은 최대한 배제하고 데이터 표시와 사용자 입력만 담당합니다.
- **ViewModel:** View와 Model 사이의 중재자입니다. "오늘 지출만 필터링하기", "금액 합산하기" 등의 핵심 로직이 위치합니다.
- **SwiftData:** 애플의 최신 DB 프레임워크로, 서버 없이 맥북/아이폰 로컬에 데이터를 안전하게 저장합니다.

### **[3주차] 핵심 로직 (CRUD & 연산)**

- **A님 작업:** 입력 시트(`AddExpenseView`) 구현 및 데이터 저장 로직 개발.
- **B님 작업:** 저장된 데이터를 불러와 리스트에 바인딩하고, 당일 합계를 구하는 연산 로직 개발.
- **공통:** 서로의 코드를 리뷰하며 데이터가 제대로 저장되고 불러와지는지 확인.

### **[4주차] 고도화 및 배포 (Polish & Test)**

- **공통 작업:** * 빈 화면일 때 "첫 지출을 기록해보세요!" 안내 문구 넣기.
    - 앱 아이콘 디자인 및 적용.

## 7. 협업 가이드 (Developer's Rule)

- **Branch 전략:** 기능을 만들 때마다 `feature/add-view`, `feature/list-view` 식으로 브랜치를 나눠 작업하세요.
- **Conflict 해결:** 코드가 겹쳐서 충돌이 나면, 주말 세션에 맥북을 같이 보면서 해결하는 것을 추천합니다. (이 과정에서 실력이 가장 많이 늡니다.)
